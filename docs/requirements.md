# 要件定義書

## 1. 背景

シロアリ防除・点検の訪問販売業務では、現在ゼンリンの紙地図コピーに施工済み、点検予定、訪問済み、不在、訪問NGなどの情報を手書きで管理している。

営業担当者は9人、1人あたり紙地図100枚超を扱う想定である。紙地図は探す、めくる、転記する、コピー差し替えする、担当者間で共有する作業が重く、訪問先確認のため徒歩1万歩超になるケースもある。

既存の紙地図スキャンやコピー画像をアプリへ取り込む設計にはしない。地図データは正規ライセンスのAPI利用を前提とし、住宅地図レベルの精度が必要な場合はZENRIN等の法人契約を検討する。

## 2. 目的

- 紙地図運用をデジタル化し、訪問情報を共有できる状態にする
- 担当者間の重複訪問、訪問NG先への誤訪問を減らす
- 定期点検、再訪問、見込み、契約、施工済みの状態を地図で確認する
- 訪問予定を作成し、ルート最適化で移動負担を減らす
- 営業担当者の徒歩負担、地図確認負担を減らし、訪問予定時間を増やす
- 管理者が担当者別・ステータス別の状況を把握できるようにする
- 未決定理由をノウハウとして蓄積し、対策案の優先度づけ、採用 / 未採用判断、営業担当者への周知に使えるようにする

## 3. 対象ユーザー

- 管理者
- 営業担当者

## 4. MVP対象範囲

### 4.1 ログイン

- 管理者、営業担当者のロールを切り替えられる
- ログアウトできる
- MVPではモック認証とし、将来Cognitoへ差し替える

### 4.2 ダッシュボード

- 管理者は全体件数、担当者別件数、ステータス別件数、重複候補、要対応データを確認できる
- 管理者は営業KPI、未決定理由ナレッジ、システム管理へ画面上部のショートカットからすぐ移動できる
- 営業担当者は自分の担当エリア、自分の訪問予定、未訪問、再訪問、点検予定を確認できる
- 営業担当者は地図、地点入力、訪問予定へ画面上部のショートカットからすぐ移動できる

### 4.3 地図画面

- 地点ピンをステータス別に色分けして表示する
- 検索、ステータスフィルタ、担当者フィルタを使える
- 地点を選択すると詳細パネルを表示する
- 地図タップまたは住所入力で地点の位置を計算できる
- 地図を全画面表示できる
- 画面上部のショートカットから地図画面へ移動できる

### 4.4 地点管理

- 地点を追加、編集、論理削除できる
- 顧客名、住所、ステータス、担当者、施工日、最終訪問日、次回点検予定日、メモ、タグを管理する
- 緯度経度を人が直接入力しなくてもよいようにする

### 4.5 訪問履歴

- 地点ごとに訪問履歴を登録できる
- 訪問結果に応じて地点ステータスと最終訪問日を更新する
- 次回アクション日、メモを記録する

### 4.6 手書きメモ

- 地点詳細に手書きメモを追加できる
- Canvasでペン、消しゴム、クリア、保存を使える
- MVPではブラウザ保存、将来S3保存へ差し替える

### 4.7 訪問予定

- 訪問日ごとに訪問予定を作成できる
- 地図上のピンを複数選択して一括追加できる
- 180件規模の候補でも、詳細を1件ずつ開かずに選べるUIにする
- 今日の作業サマリーで訪問先数、次の訪問先、ルート状態、選択中件数を確認できる

### 4.8 ルート最適化

- MockRouteServiceで訪問順、総距離、推定時間、地図上ルート線を表示する
- 将来Amazon Location Serviceまたは他のルートAPIへ差し替える

### 4.9 重複・訪問NG警告

- 同住所、正規化住所一致、半径20m以内、顧客名類似を検知する
- 施工済み、点検予定、訪問NGを警告する
- MVPでは登録を止めず、判断材料として表示する

### 4.10 CSV入出力

- 地点データをCSVエクスポートできる
- CSVインポート時、住所のみでもMockGeocodingServiceで座標を補完する

### 4.11 未決定理由・対策ナレッジ

- 営業担当者は地点詳細から未決定理由を入力できる
- 未決定理由にはリスク、対策案、優先度を紐づける
- 管理者は未決定理由を一覧で確認できる
- 管理者はJAエリア、市町村、優先度、判断状況で絞り込み・集計できる
- 管理者は対策案に対して採用 / 検討中 / 未採用を記録できる
- 未採用の場合は未採用理由を記録できる
- 採用した対策は営業担当者画面へ周知できる
- 登録、管理者更新、周知判断はログとして残す
- 仮説検証の注釈を画面に表示する。仮説は効果未確定のため複数出し、対策案は優先度を決めて逐次実行し、実行結果を計測して定期的に見直す

## 5. 非機能要件

- タブレット幅でも操作できる
- 個人情報を含むため、本番ではCognito認証と権限制御を必須にする
- 削除は論理削除にする
- AWS移行を前提にRepository層、MapProvider、GeocodingService、RouteServiceを分離する
- 初期は厳密なオフライン対応を対象外とするが、将来対応を妨げない

## 6. コスト・効果試算の前提

- 営業担当者9人、管理者1〜3人
- 人件費は1人月60万円、1人月160時間、1時間3,750円で仮置き
- 現行非効率は1人あたり月16〜30時間
- 導入後は1人あたり月3〜6時間まで削減できると仮定
- 人件費換算の改善効果は月33.75万〜84.375万円
- 削減時間の70%を訪問予定時間に回せると、月126〜315件分の訪問予定枠に相当
- AWS等の小規模サーバーレス費用は月1万〜6万円程度を仮置き
- 住宅地図API、商用地図APIは別途見積り
- 折り畳み自転車、スマホ・タブレットホルダー等で徒歩負担軽減も検討する

## 7. 受け入れ条件

- ログイン画面から管理者、営業担当者として入れる
- 管理者と営業担当者で表示内容が切り替わる
- 地図上に地点ピンが表示され、検索・フィルタできる
- 住所入力または地図タップで地点を追加できる
- 地点を編集、論理削除できる
- 訪問履歴を追加できる
- 手書きメモを保存・表示できる
- 訪問予定を作成し、地図から複数地点を追加できる
- ルート最適化を実行できる
- 重複候補、訪問NG候補を警告できる
- CSVインポート、エクスポートができる
- 営業担当者が未決定理由と対策案を入力できる
- 管理者が未決定理由をエリア別・市町村別・優先度別に確認できる
- 管理者が採用 / 未採用理由を記録し、採用済み対策を営業担当者へ周知できる
- ヘッダーショートカットで営業担当者は地図・入力・訪問予定へ、管理者はKPI・未決定理由・地図・入力・設定へ移動できる
- TypeScriptチェック、lint、テスト、ビルドが通る
